# 振り返り・自己改善ルール

壁打ちのやり方を継続的に改善するためのルール。対話ログからの気づきを `sparring.md` に反映していく仕組み。

## 対話中の気づきを記録する

壁打ちセッション中、以下のような「シグナル」を検知したら `observations.md` に記録する。ユーザーへの確認は不要、静かに記録する。

### ポジティブシグナル（うまくいった）

- ユーザーが「いいね」「しっくりきた」「鋭い」と反応した直前のアプローチ
- ユーザーが自分の現場の話を自発的に話し始めたきっかけ
- ユーザーが「なるほど」と納得して次に進んだパターン

### ネガティブシグナル（うまくいかなかった）

- ユーザーが「違う」「もっと◯◯して」と軌道修正を求めたケース
- ユーザーの指摘で自分の説明が不正確だったと気づいたケース
- 説明が長すぎて「要約して」と言われたケース
- 質問の意図を取り違えたケース

### 記録フォーマット（observations.md）

```markdown
## YYYY-MM-DD ◯章

### うまくいった
- 〜のときに〜した。ユーザーの反応：〜

### うまくいかなかった
- 〜してしまった。ユーザーの指摘：〜。学び：〜
```

## レトロスペクティブ

ユーザーが「振り返りして」「retro」と言ったら、直近の `observations.md` を読んで以下を提示する。

1. 今回のセッションで効いたアプローチ
2. 改善したいパターン
3. `sparring.md` への反映案（具体的な文言修正）

ユーザーの承認を得たら `sparring.md` を更新する。更新後は該当する `observations.md` のエントリを「反映済み」としてマークする。

## sparring.mdの自己更新ルール

### 更新してよいケース

- 複数回同じパターンの気づきが記録されている（1回だけの偶発的な出来事では更新しない）
- ユーザーが明示的に「このルール追加して」と言った
- レトロスペクティブでユーザーが承認した

### 更新してはいけないケース

- 1セッション1回だけの気づき
- ユーザーの承認なしの自己判断での更新
- 既存ルールと矛盾する内容（矛盾を検知したらユーザーに確認）

### 更新時の作法

- 元のルールを残したまま追記する形を基本とする
- 大きな方針変更は必ずユーザーに確認
- 変更履歴を `sparring.md` 末尾の `## 更新履歴` セクションに残す

## observations.mdのメンテナンス

- `sparring.md` に反映済みのエントリは3ヶ月を目安に削除してよい
- 反映されていないエントリは残す（後から効くパターンかもしれない）
