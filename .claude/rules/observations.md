# 対話ログからの気づきメモ

壁打ちセッション中の気づきを記録する。レトロスペクティブ時に参照してsparring.mdに反映する。

## 記録例

```
## YYYY-MM-DD ◯章

### うまくいった
- 〜のときに〜した。ユーザーの反応：〜

### うまくいかなかった
- 〜してしまった。ユーザーの指摘：〜。学び：〜
```

---

## 2026-04-16 第三部SOLID・第四部12-14章

### うまくいった

- 自分の実装コード（BaseService）を題材にしたらエンジン全開になった
- 「本の主張」と「現実のトレードオフ」を対比する説明（OCPの「変更ゼロは無理」など）
- 悪い例→良い例の順でコードを示す
- ユーザーの疑問をまず肯定してから補足・訂正する
- 1回の返答で1〜2ポイントに絞る

### うまくいかなかった

- 最初のOCPまとめでユーザー自身の指摘が反映されていなかった → 「自分の話が反映されていない」と指摘された。学び：まとめはユーザーが実際に言ったことを軸にする
- コンポーネントの説明で「デプロイできる単位」と言ったら違和感を持たれた → 用語の定義を曖昧にしない。デプロイとリリースの違いなど、言葉の境界を揃える
- 「Recordだけ切り出しても意味がない」と言い切った → ユーザーに「リリースを小さくするなら分けてもいい」と指摘され訂正。学び：言い切り型の主張をする前に、トレードオフを確認する

### 観察

- ユーザーは「それってメタプロ？」のような類似概念との比較質問が多い。SOLIDや設計原則は「似てるけど別物」の対比で説明すると腑に落ちやすい
- 用語の違和感にこだわるタイプ。「デプロイとリリースの違いは？」のような質問が出る
- 「ok」「次」で切り上げを示す。詰め込みすぎを嫌う
